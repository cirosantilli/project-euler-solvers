# Project Euler 660 — Pandigital Triangles (solution summary)

A triangle with a `120°` angle and integer sides can be labelled `(a, b, c)` so that the `120°` angle is opposite `c`.
By the Law of Cosines:

\[
c^2 = a^2 + ab + b^2
\]

The problem asks for all `n`-pandigital such triangles for `9 ≤ n ≤ 18`, where “pandigital” means:
when `a`, `b`, `c` are written in base `n`, **their digits together use each digit `0..n-1` exactly once**.

---

## Key techniques used

### 1) Turn geometry into a Diophantine constraint
The 120° condition becomes the exact integer equation:

\[
a^2 + ab + b^2 - c^2 = 0
\]

Also, since `120°` is the largest angle, `c` is automatically the largest side.

---

### 2) Work in base `n` digit-by-digit with carry propagation
Write `a`, `b`, `c` in base `n`:

\[
a=\sum a_i n^i,\;\; b=\sum b_i n^i,\;\; c=\sum c_i n^i
\]

Define:

\[
E = a^2 + ab + b^2 - c^2
\]

Process digits from least significant upward. For each position `t`, compute the coefficient of `n^t` in `E` and maintain a carry so that after processing position `t` we ensure:

- the partial equation holds modulo `n^{t+1}`,
- i.e. the “current coefficient + carry” is divisible by `n`.

This is analogous to doing long arithmetic in base `n`, except applied to the polynomial identity above.

---

### 3) Use the fact that, for `t > 0`, the constraint is *linear* in the new digits
At position `t>0`, the coefficient depends on `a_t`, `b_t`, `c_t` only via products with the least significant digits `a_0`, `b_0`, `c_0`.
That makes the modular constraint at each step a **linear congruence**:

\[
(\text{known}) + (2a_0+b_0)a_t + (a_0+2b_0)b_t - 2c_0 c_t \equiv 0 \pmod n
\]

This lets the solver precompute residue tables and avoid brute forcing all triples every time.

---

### 4) Bitmask state for “each digit used exactly once”
Digits are `0..n-1` with `n ≤ 18`, so an `int` bitmask is perfect:

- bit `d` set ⇒ digit `d` is still available
- choosing a digit clears the bit

This makes “pandigital” enforcement extremely cheap.

---

### 5) Strong pruning from triangle size
With `a < b` and `c < a+b ≤ 2b`, the base-`n` digit length of `c` cannot exceed that of `b` by more than 1:

- only consider length splits with `lc ≤ lb + 1`
- if `lc = lb + 1`, then `c` lies in `[n^lb, 2·n^lb)`, so the most significant base-`n` digit of `c` must be `1`

This removes a huge amount of impossible search.

---

## What the program outputs
Running `python3 main.py` prints the required sum for `9 ≤ n ≤ 18`.

The code includes `assert`s for the explicit example triangle in the statement:
`(217, 248, 403)` is `9`-pandigital and satisfies the 120° equation.
