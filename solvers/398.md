# Project Euler 398 — Cutting Rope (technique summary)

We pick `m-1` cut points uniformly from the `n-1` integer points inside the rope.  
That is equivalent to picking a **uniform random ordered composition**:

- segment lengths are positive integers `x1, …, xm`
- `x1 + … + xm = n`
- each ordered `m`-tuple occurs exactly once for some choice of cut points (`stars and bars`), so all are equally likely.

## 1) Turn “expected value” into probabilities (tail-sum)

Let `Y` be the **second-shortest** segment length (ties allowed). For integer-valued `Y`:

\[
E[Y] = \sum_{k\ge 1} P(Y \ge k)
\]

So the whole task becomes computing `P(Y ≥ k)` efficiently.

## 2) Interpret the event `Y ≥ k`

`Y ≥ k` means **fewer than two segments are shorter than `k`**:

- either all `m` segments have length `≥ k`
- or exactly one segment has length `< k` and the other `m-1` have length `≥ k`

That “at most one short segment” characterization is the key simplification.

## 3) Count the valid compositions with binomial coefficients

Using stars-and-bars:

- The number of compositions of `N` into `m` positive integers is `C(N-1, m-1)`.
- The number of solutions in nonnegative integers is also a binomial coefficient.

For fixed `k`, the count of compositions where `Y ≥ k` can be written in terms of binomials and then simplified with a **hockey-stick identity** to:

\[
\text{count}(Y \ge k) = m\binom{N_2}{m-1} - (m-1)\binom{N_1}{m-1}
\]

where

- \(N_2 = n - (m-1)k + (m-2)\)
- \(N_1 = n - mk + (m-1)\)

The total number of compositions is \(\binom{n-1}{m-1}\), so:

\[
P(Y \ge k) = \frac{\text{count}(Y \ge k)}{\binom{n-1}{m-1}}
\]

Also note the sum only needs to run up to `k = floor((n-1)/(m-1))`, because otherwise even `m-1` segments of length `k` wouldn’t fit.

## 4) Avoid huge integers: compute *ratios* of binomials

For the real target (`n=10^7`, `m=100`) the binomial coefficients are enormous.

Instead of computing them directly, we compute ratios like:

\[
\frac{\binom{A}{r}}{\binom{B}{r}} = \prod_{i=0}^{r-1} \frac{A-i}{B-i}
\]

This stays in normal floating-point range and is accurate enough for rounding to 5 decimals.

## Complexity

Let `r = m-1`. We sum over about \((n-1)/(m-1)\) values of `k` (≈ 101k for the target), and each probability term uses `O(r)` multiplications (here `r=99`).

Overall work is on the order of **~10 million float ops**, which is fast in Python.
