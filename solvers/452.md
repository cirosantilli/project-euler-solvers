# Project Euler 452 — Long Products (Python)

## Problem
Define `F(m, n)` as the number of `n`-tuples of positive integers whose product does not exceed `m`.

Given:
- `F(10, 10) = 571`
- `F(10^6, 10^6) mod 1234567891 = 252903833`

Find:
- `F(10^9, 10^9) mod 1234567891`

---

## Core Insight: Only ~log2(m) Entries Can Be > 1

If the tuple product is ≤ `m`, and each non-1 entry is at least `2`, then:

```
2^k ≤ m
k ≤ floor(log2(m))
```

So for `m = 10^9`:

```
floor(log2(10^9)) = 29
```

That means **in any valid tuple, at most 29 entries differ from 1**, even though `n = 10^9`.

This transforms the problem from “count 10^9-tuples” into “count small multisets of factors”.

---

## Pattern Enumeration (Multiplicity States)

Instead of working with full tuples, the solution counts **configurations** of numbers > 1:

Example configuration:
- `a` appears 10 times
- `b` appears 2 times
- `c` appears 1 time

This corresponds to the constraint:

```
a^10 * b^2 * c ≤ m
```

We represent such configurations by a **pattern**:

```
(10, 2, 1)
```

We enumerate all patterns that can occur under the product constraint and count how many distinct sets of values realize each pattern.

### Pruning Trick
During recursion, if the remaining limit is smaller than `min_val²`, then:
- you **cannot** pick two distinct numbers ≥ `min_val`
- you **cannot** pick exponent ≥ 2

So you can only add **at most one more value once** — and that entire range can be counted in O(1).

---

## Counting Ordered Tuples From a Pattern

Once we know a pattern’s count, we compute how many ordered `n`-tuples it produces.

If the pattern sums to `k` non-1 entries:

```
ways = n! / (n-k)! / Π(mi!)
     = falling_factorial(n, k) / Π(mi!)
```

- `falling_factorial(n, k) = n*(n-1)*...*(n-k+1)`
- Divide by `mi!` because repeated values are indistinguishable

### Composite Modulus Handling
`1234567891` is not prime, so modular inverses are unsafe.

Instead, the solution:
- builds the numerator factors explicitly (`k ≤ 29`)
- cancels denominator factorials via `gcd()` in integers
- then takes modulo at the end

This keeps the method exact and robust.

---

## Complexity
- Pattern recursion is small because:
  - depth ≤ ~11 distinct values
  - total multiplicities ≤ 29
  - heavy ranges are compressed using pruning

The number of unique patterns is small (on the order of a few thousand), making the final summation fast.
