# Project Euler 426 — Box-Ball System (solution approach)

This problem asks for the long-time **final state** of a Box–Ball System (BBS), starting from a very large run-length encoded configuration, and then for the **sum of squares** of the occupied block sizes in that final state.

## Key idea: solitons are invariant

A standard fact about the BBS is that after enough turns, the configuration separates into **solitons**: blocks of consecutive balls that move without changing shape. The **multiset of soliton sizes** is conserved by the dynamics, and in the separated regime the occupied blocks (from left to right) are exactly these soliton sizes in increasing order.

So we don’t simulate turns. We only need to compute the soliton sizes once from the initial configuration.

## Takahashi–Satsuma “data stream” algorithm

We use the Takahashi–Satsuma soliton identification algorithm in a streaming form:

1. Start with an infinite run of `0` (empty boxes) to the left.
2. Read the configuration left-to-right, appending one symbol at a time.
3. Whenever the **last two runs** have the **same length**, those runs annihilate.
   - Each annihilation corresponds to **one soliton**, and the soliton size equals that run length.

Accumulating `size²` for every annihilation directly gives the required answer (sum of squares of the final occupied blocks).

## Run-length acceleration (no expansion)

The input is already given as alternating run lengths `(t0, t1, t2, ...)` of `1`s and `0`s. Expanding to individual boxes would be far too large.

Instead we maintain a **stack of runs** (symbol + length) with an important invariant:

- Run lengths on the stack are **strictly decreasing** from bottom to top.

Appending a whole run of length `L` can trigger multiple annihilations, but we can process it in chunks:

- If we would grow a run to match the previous run’s length, we “stop at equality”, annihilate, record a soliton, and continue with any leftover length.

This gives an **amortized linear** algorithm in the number of input runs.

## Complexity

- Let `N` be the number of run-length terms (here `N = 10,000,001`).
- Time: `O(N + A)` where `A` is the number of annihilations (solitons); in practice this runs quickly in Python.
- Memory: `O(D)` where `D` is the current stack depth (small compared to expanded configuration length).
