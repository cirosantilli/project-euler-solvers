# Project Euler 386 — Optimization Notes

This solution computes:

\[
\sum_{n \le 10^8} N(n)
\]

where \(N(n)\) is the maximum size of an **antichain** in the divisor set \(S(n)\) ordered by divisibility.

## Main techniques

### 1) Divisors form a ranked product-of-chains poset
If:

\[
n = \prod_i p_i^{a_i},
\]

then each divisor corresponds to an exponent vector \((e_i)\) with \(0 \le e_i \le a_i\), and divisibility is coordinate-wise \(\le\).

The poset is ranked by:

\[
\Omega(d) = \sum_i e_i.
\]

The number of divisors of rank \(k\) is the coefficient of \(x^k\) in:

\[
\prod_i (1 + x + x^2 + \dots + x^{a_i}).
\]

### 2) Largest antichain = largest rank layer
Products of chains have the Sperner property: a maximum antichain is achieved by taking a largest rank layer.

The generating polynomial above is **palindromic**, so the maximum layer occurs at:

\[
k = \left\lfloor \frac{\sum a_i}{2} \right\rfloor.
\]

So \(N(n)\) is simply the coefficient of that middle term.  
`main.py` computes it with a tiny DP using a sliding-window convolution.

### 3) Avoid iterating all \(n \le 10^8\)
Instead of factoring every integer up to \(10^8\), we enumerate **exponent patterns**:

\[
n = p_1^{e_1} p_2^{e_2} \dots p_k^{e_k}, \quad p_1 < p_2 < \dots < p_k,
\]

and for each pattern:

1. Count how many \(n \le \text{limit}\) realize that pattern.
2. Multiply by the corresponding \(N(\cdot)\) (depends only on the multiset of exponents).

For \(\text{limit}=10^8\), \(k \le 8\).

### 4) Fast counting via recursion with a strong bound
When choosing the next prime \(p\) at recursion position `pos`, the remaining exponents sum to \(S\).  
Because the remaining primes are all \(\ge p\), we must have:

\[
p^S \le \text{remaining\_limit}.
\]

So:

\[
p \le \left\lfloor \sqrt[S]{\text{remaining\_limit}} \right\rfloor,
\]

which keeps prime loops tiny.

At the final level we need “how many primes are in \((p_{\text{prev}}, p_{\max}]\)”, which reduces to prime counting.

### 5) Faster prime counting for this problem: sieve once + bisect
The main speed improvement (vs. a heavy prime-counting algorithm) is:

- Build all primes up to `limit` using an **odd-only sieve** (bytearray + slicing).
- Answer \(\pi(x)\) queries using `bisect_right(primes, x)`.

For \(10^8\), there are about 5.76 million primes, which fits comfortably in an `array('I')` and makes \(\pi(x)\) very fast.
