# Project Euler 662 — Fibonacci Paths (Python)

This repository contains a clean, dependency-free Python implementation for **Project Euler 662**.

## Problem recap

Alice moves on the integer lattice from `(0,0)` to `(W,H)`.  
A step from `(a,b)` to `(a+dx, b+dy)` is allowed if:

- `dx ≥ 0` and `dy ≥ 0`
- The Euclidean step length is a Fibonacci number:

\[
\sqrt{dx^2 + dy^2} \in \{1,2,3,5,8,13,\dots\}
\]

Let `F(W,H)` be the number of valid paths.  
We want `F(10000,10000) mod 1_000_000_007`.

The statement also provides test values:

- `F(3,4) = 278`
- `F(10,10) = 215846462`

## Main techniques used

### 1) Step generation from Fibonacci circles

For every Fibonacci number `f ≤ max(W,H)`, we generate all lattice points on the quarter-circle:

\[
dx^2 + dy^2 = f^2
\]

We do this by iterating `dx` from `0..f` and checking whether `f^2 - dx^2` is a perfect square.

This yields a **small finite step set** (about ~80–100 step vectors for `f ≤ 6765`).

### 2) Dynamic Programming (DP) on the grid

Let `dp[x][y]` be the number of ways to reach `(x,y)`.

Then:

\[
dp[x][y] = \sum_{(dx,dy)\in S} dp[x-dx][y-dy]
\]

where `S` is the allowed step set and invalid indices are ignored.

The base case:

- `dp[0][0] = 1`

### 3) Rolling row buffer (memory optimization)

A naive DP table for `(10000,10000)` has ~100 million cells, which is too large for Python lists.

We observe that to compute row `y`, we only need rows:

\[
y - dy
\]

for `dy` in the step set, so we only need to keep the last:

\[
max(dy) + 1
\]

rows in memory.

This reduces memory from **O(W·H)** to:

\[
O(W·max\_dy)
\]

which is manageable.

## Notes about performance

This implementation is a correct baseline and passes the sample asserts, but computing `(10000,10000)` is still extremely heavy in pure CPython because it involves large-scale convolution-like DP.

To make it fast enough for Euler-scale execution, the next step typically involves:

- diagonal extraction methods
- D-finite / holonomic recurrences
- constant-term / residue techniques
- or fast polynomial operations (NTT/CRT)

Those are not implemented here, but the code is structured so that such improvements can replace the DP core cleanly.
