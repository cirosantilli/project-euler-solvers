# Project Euler 663 — Sums of Subarrays

This repository contains a standalone `main.py` (no third‑party dependencies) that solves Project Euler **Problem 663** and checks all sample values from the statement with `assert`s.

## Key ideas used

### 1) Split the simulation into two phases
We need

\[
S(n,10\,200\,000) - S(n,10\,000\,000) = \sum_{i=10\,000\,001}^{10\,200\,000} M_n(i).
\]

The maximum subarray sum **after step _i_** depends on all earlier updates, but we do **not** need to *query* it during the first `10,000,000` steps.

So we:

1. Apply the first `10,000,000` updates directly to the array `A` (cheap `O(1)` point adds).
2. Build a data structure once from that final array.
3. Process only the last `200,000` updates while querying `M_n(i)` each time.

This removes the need for `10,000,000` expensive “update + query” operations.

### 2) Fast generation of tribonacci values modulo `n`
Only `t_k mod n` is required. Since every term is already reduced, the next value

\[
t_{k+1} \equiv t_k + t_{k-1} + t_{k-2} \pmod n
\]

lies in `[0, 3n)`, so modulo can be replaced with a few conditional subtractions (faster than `%`).

### 3) Max subarray sum via block summaries
To support point updates and quickly read the global maximum subarray sum, the array is split into fixed blocks (default `B = 128`).

For each block we store the classic 4-tuple:

- `tot`  — sum of the block
- `pref` — best (max) prefix sum within the block
- `suff` — best (max) suffix sum within the block
- `best` — best (max) subarray sum within the block

These are computed using prefix sums + Kadane’s algorithm.

### 4) Segment tree over blocks
The block tuples are combined with a segment tree where two adjacent segments `L` and `R` merge as:

- `tot  = totL + totR`
- `pref = max(prefL, totL + prefR)`
- `suff = max(suffR, totR + suffL)`
- `best = max(bestL, bestR, suffL + prefR)`

This makes the global `M_n(i)` simply `best` at the tree root.

### 5) Cheap updates during the queried range
Each of the final `200,000` steps performs:

1. A single point add to `A[idx]`.
2. Recompute *only the affected block* in `O(B)`.
3. Update `O(log #blocks)` segment tree nodes.

With `B = 128`, the per-step work is small enough to run comfortably in Python.

## Complexity

Let `n = 10,000,003` and `B = 128`.

- Phase 1 (first 10M updates): `O(10,000,000)` point adds.
- Build: `O(n)` to compute all block summaries + `O(#blocks)` to build the tree.
- Phase 2 (200k queried updates): `O(200,000 * (B + log #blocks))`.

Memory is dominated by storing the array `A` (64-bit integers).
