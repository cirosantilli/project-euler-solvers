# Project Euler 665 – Proportionate Nim (technique summary)

This solution computes

- **losing (P) positions** of the two-pile game, and
- `f(M)`: the sum of `n+m` over all losing positions `(n,m)` with `n ≤ m` and `n+m ≤ M`.

The required value is `f(10^7)`.

## Key idea: greedy generation of P-positions

The legal moves remove stones by vectors that are integer multiples of:

- `(1,0)` or `(0,1)`  (remove from one pile),
- `(1,1)`            (remove equally from both),
- `(1,2)` or `(2,1)` (remove proportionally).

For an **invariant subtraction game**, the standard way to build P-positions is:

1. Let `a` be the **mex** (minimum excluded) of all pile sizes already used in any P-position.
2. Choose the smallest `b ≥ a` that does **not** collide with any move-invariant of a previous P-position.

Here the necessary uniqueness constraints are:

- `a` and `b` must be new values (otherwise a single-pile move connects two P-positions).
- `d = b-a` must be new (otherwise a diagonal move `(k,k)` connects them).
- `t1 = b-2a` must be new (otherwise a `(k,2k)` move connects them).
- `t2 = a-2b` must be new (otherwise a `(2k,k)` move connects them).

This greedy mex procedure generates exactly the losing positions.

## Turning the constraints into fast checks

Three data structures are maintained:

- **used numbers** (all `a` and `b`),
- **used differences** (`b-a`),
- **used linear values** (`b-2a` and `a-2b`).

To make the search for `b` fast:

- `used numbers` and `used differences` are implemented as **disjoint-set “next free”** tables.
  - `find(x)` returns the smallest unused value ≥ `x`.
  - This avoids repeated linear scanning in dense regions.
- `used linear values` are stored in a `bytearray` over the integer range `[-2L, L]`
  (shifted to nonnegative indices).
- The constraint `a-2b` is rewritten as `u = 2b-a` (which increases by `2` when `b` increases).
  - `u` is tracked in two parity tables (even/odd), allowing fast “next unused with same parity”.

## Counting `f(M)`

While generating P-positions, we add `a+b` to the answer when `a+b ≤ M`.

We only need to generate positions with `a ≤ ⌊M/2⌋` because `b ≥ a` implies `a+b > M` once `a` exceeds `M/2`.

## Complexity

- Time: roughly linear in the number of generated P-positions (a few million for `M=10^7`).
- Memory: a handful of large compact arrays (bytearrays + two integer DSU tables).
