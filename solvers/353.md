# Project Euler 353 — Risky Moon (Solution Notes)

## Core idea

Stations are all integer-coordinate points on the sphere:

\[
x^2 + y^2 + z^2 = r^2.
\]

Every pair of stations is connected by the **shortest great-circle arc**.
If the central angle between two radius vectors is \(\theta\), the road risk is:

\[
\left(\frac{\theta}{\pi}\right)^2.
\]

We need the **minimal total risk** from the North Pole \((0,0,r)\) to the South Pole \((0,0,-r)\).

---

## Key techniques used

### 1) Convert great-circle distance to chord length
For two points on a sphere of radius \(r\):

- Euclidean chord length: \(d = \lVert p-q \rVert\)
- Central angle:

\[
\theta = 2\arcsin\left(\frac{d}{2r}\right)
\]

Thus risk can be computed purely from **integer** chord-squared:

\[
s = d^2,\quad
\text{risk}(s)=\left(\frac{2\arcsin(\sqrt{s}/(2r))}{\pi}\right)^2.
\]

This means edge weights depend only on the squared chord distance \(s\).

---

### 2) Implicit graph instead of complete graph
The true graph is complete (every station connects to every other), which is far too large.

Instead, we compute the answer by building a **sparse implicit graph**:

- A node is any integer point on the sphere.
- Edges are only explored to points whose chord-squared distance \(s\) is ≤ some limit `smax`.

The neighbor generation is done without precomputing all stations.

---

### 3) Neighbor generation by scanning \((x,y)\) offsets
From a station \((x,y,z)\), we generate nearby candidates by:

1. Add small integer offsets \((dx,dy)\) to get \((x',y')\).
2. Solve for \(z'\) using the sphere equation:

\[
z'^2 = r^2 - x'^2 - y'^2
\]

3. If \(z'^2\) is a perfect square, we have 1 or 2 valid candidates (\(z'=\pm\sqrt{z'^2}\)).

This finds all stations reachable with the chosen local window efficiently.

---

### 4) Bidirectional Dijkstra
All edge weights are positive, so shortest path is solved using Dijkstra.

To reduce explored states, a **bidirectional** variant is used:

- One search from North Pole.
- One search from South Pole.
- Stop when frontiers cannot improve the best meet-in-the-middle connection.

---

### 5) Progressive radius expansion (`smax`)
We do not know a priori how large `smax` must be.

So we:

1. Start with a small `smax` (local edges only).
2. Double until a path exists.
3. Keep doubling while the best value improves meaningfully.

This approach is practical and converges quickly in practice for the specific
radii required by the problem.
